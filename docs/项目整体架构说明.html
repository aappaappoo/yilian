Soulbot / Soulmeet 全局架构

项目整体架构说明

这个项目不是单纯的 Agent Loop。它是一套“前端陪伴体验 + 后端会话运行时 + Soul companion Agent 工具能力 + 语音链路 + 记忆系统 + 部署服务”的组合架构。

一句话看懂

前端负责陪伴式聊天界面、3D 角色、文字/语音输入和状态展示；后端负责会话、认证、记忆、语音管线和 Agent 工具调用；真正回答用户生活查询的是 core/soul_companion 里的 Soul Agent，它通过 LLM tool-calling 调用天气、车票、餐厅、景点、健康、提醒、联网搜索等工具。

可以把它理解成四层：用户体验层、API 接入层、会话智能层、基础设施层。Agent Loop 只是“会话智能层”里的一个子模块，不是整个项目。

总览图

用户体验层
apps/web：Vue 3 + Vite 单页应用。包含首页、聊天页、桌面/移动聊天布局、Aini 角色、3D VRM、语音按钮、快捷问题、天气标记、会话列表、登录注册等。
↓ HTTP / SSE / LiveKit / WebRTC
API 接入层
apps/backend/core/api：FastAPI 路由。包括文本问答、实时语音、按住说话识别、会话列表、用户认证、角色资源、天气栏、快捷建议、声音克隆等接口。
↓ 统一进入会话运行时
会话智能层
ConversationSessionManager 管理同一个业务会话。文字输入和语音转写后的文本都会进入这里，然后调用 answer_user_text，再进入 run_soul_agent 执行工具调用循环。
↓ 工具 / 记忆 / 外部服务
能力与基础设施层
plugins/soul-companion 提供生活查询工具；AsyncContextManager、SQLStore、MemoryFileStore 提供记忆；DashScope、DeepSeek、高德、12306、LiveKit、PostgreSQL/pgvector、coturn 等作为外部依赖。

前端架构

启动入口

apps/web/src/main.ts 创建 Vue App，挂载 Pinia、Router、i18n、Sentry、UnoCSS 和全局主题。

路由

apps/web/src/router/index.ts 有首页 /、默认聊天页 /chat、角色聊天页 /chat/:audience、指定对话页 /chat/:audience/:conversationId。

聊天总控

apps/web/src/pages/chat.vue 是核心页面，组合聊天消息、3D 角色、语音连接、文本请求、SSE 进度、快捷建议、天气、会话列表、登录弹窗等。

布局组件

DesktopChatLayout.vue 和 MobileChatLayout.vue 分别负责桌面和移动端聊天界面。

状态管理

stores/session.ts 保存连接状态、当前角色、情绪状态、当前节点、可用工具；stores/auth.ts 保存登录态和 token。

实时语音

composables/useWebRTC.ts 根据配置使用 LiveKit 或自建 WebRTC，处理音频轨道、数据消息、语音模式、VAD 中断设置。

聊天消息

composables/useChat.ts 管理本地消息、流式消息、后端进度事件、任务过程展示。

3D 角色

@soulmeet/stage-three 导出 AvatarCanvas，聊天页按需加载 VRM 角色，支持禁用回退。

后端架构

后端模块 职责
main.py 应用启动：配置日志、CORS、全局单例、数据库、记忆系统、路由、提醒调度器。
core/api HTTP API 层：文本、语音、认证、会话、角色、天气、快捷建议、声音克隆、WebRTC/LiveKit。
core/conversation 会话与记忆层：会话运行时、历史、SQLStore、文件记忆、异步上下文分析、embedding。
core/soul_companion 文本智能核心：Soul runtime、Agent Loop、默认地点、提醒调度、提示词。
core/pipeline 实时语音管线：Transport 输入 → 音频记录 → DashScope STT → 情绪检测 → Soul 回复 → DashScope TTS → Transport 输出。
core/auth 用户注册、登录、JWT、API Key、用户身份依赖。
core/persona 读取角色 persona.yaml，给语音、角色资源和人设配置使用。
core/tools 基础工具注册表和记忆工具。当前 Soul 生活查询主要由 soul-companion 插件直接提供。

文字聊天主链路

  1. 用户在前端聊天页输入文字。
  2. chat.vue 打开 /api/text-runtime/events/{session_id} 的 SSE，用来接收“正在判断、正在查询、正在整理”等过程事件。
  3. chat.vue 调用 POST /api/text-runtime/message 发送文本、角色、session、conversation 和 token。
  4. text_runtime.py 进入 HTTP 文本入口，确保 conversation 存在，创建或复用业务会话。
  5. ConversationSessionManager.handle_user_text 记录用户消息、读取最近上下文、解析默认地点、准备进度回调。
  6. answer_user_text 进入 Soul companion runtime。
  7. run_soul_agent 让 LLM 决定是否调用工具。如果需要实时信息，模型会返回 tool call。
  8. 后端执行工具，把工具结果回灌给模型，直到模型生成最终中文答复。
  9. 会话运行时把回复流式推送给前端，写入记忆系统，并在需要时触发语音播报。

实时语音主链路

  1. 前端 useWebRTC.ts 根据 VITE_VOICE_TRANSPORT 选择 LiveKit 或自建 WebRTC。
  2. LiveKit 模式先请求 POST /api/livekit/token，后端创建 room token，并启动机器人端 pipeline。
  3. 自建 WebRTC 模式走 POST /api/offer 做 SDP offer/answer。
  4. 后端 webrtc.py 创建 transport，再调用 build_pipeline。
  5. Pipeline 顺序是：音频输入 → 录音 → DashScope STT → 前端转写展示 → 情绪检测 → Soul companion 回复 → DashScope TTS → 音频输出。
  6. 语音转写后的最终文本也会进入同一个 Soul companion 回复逻辑，因此文字和语音共享智能核心。
另有“按住说话”模式，它不是实时语音 pipeline，而是短音频上传到 /api/speech/transcribe，识别成文字后再按文字聊天链路发送。

Soul Agent 与工具能力

core/soul_companion/agent_loop.py 是当前文本智能核心。它读取 prompt.md，拼接当前日期和默认地点，然后向 DeepSeek 或 DashScope 的 OpenAI-compatible chat/completions 接口发起请求。

Agent 暴露给模型的主要工具有：

  • soul_travel_plan：组合天气、车票、景点、美食、住宿，生成老人友好旅行规划。
  • soul_weather：高德天气。
  • soul_train_tickets：12306 车次/余票。
  • soul_local_search：高德 POI，美食、酒店、景点。
  • soul_health_query：易联健康记录。
  • soul_reminder：一次性定时提醒。
  • soul_set_default_location：记录用户确认的默认地点。
  • web_search、web_extract：联网搜索和网页正文提取。

工具的实际实现主要在 plugins/soul-companion/__init__.py。仓库里还有 apps/backend/plugins/soul-companion 的副本；当前运行时会优先从仓库根部的 plugins/soul-companion 找插件。

记忆系统

记忆系统不是同步回答用户的主链路，而是对话旁路。用户和助手消息会先进入会话 store，再由 AsyncContextManager 异步分析和落库。

短期上下文

会话运行时保留最近消息，作为下一轮 Agent 的上下文。

原始消息落盘

MemoryFileStore 会把原始对话写到 memories/sessions/...。

结构化记忆

AsyncContextManager 调用独立 memory LLM 分析摘要、事件、事实，再通过 SQLStore 写入数据库。

向量化检索

EmbeddingService 配合 PostgreSQL/pgvector 处理记忆 embedding。

角色与资源系统

角色不是写死在聊天页里，而是放在 apps/backend/audiences。每个角色目录通常包含 persona.yaml、头像、VRM 模型、背景图，Aini 还包含领域知识文档。

前端通过 /api/audiences 获取角色列表，通过 /api/audiences/{name}/assets/{filename} 和 /api/audiences/{name}/background/{filename} 读取资源。

外部服务依赖

服务 用途
DeepSeek / DashScope LLM Soul Agent 的工具调用决策和最终答复；记忆系统也可使用独立 memory LLM。
DashScope STT/TTS 实时语音识别、按住说话识别、语音合成。
高德地图 天气、IP 定位、本地 POI 查询。
12306 火车/动车/高铁车次和余票查询。
易联健康 心率、血压、血氧、体温等健康记录查询。
PostgreSQL + pgvector 用户、会话、消息、记忆、embedding 持久化。
LiveKit / coturn 实时语音连接、NAT 穿透。
Sentry 前后端错误监控，可选。

部署架构

开发环境和生产环境都围绕 Web、Backend、DB 组织。

  • web：构建前端并通过 Nginx/HTTPS 提供页面。
  • backend：运行 FastAPI，默认端口 8080。
  • db：PostgreSQL/pgvector，存储用户、会话和记忆。
  • coturn：生产环境用于 WebRTC TURN 中继。

关键部署文件是 deploy/docker-compose.dev.yml、deploy/docker-compose.prod.yml、deploy/nginx.conf 和 deploy/docker/Dockerfile.web。

阅读代码建议

  1. 先看 README.md，确认当前产品能力和已移除旧路径。
  2. 再看 apps/web/src/pages/chat.vue，理解用户交互怎么发给后端。
  3. 然后看 apps/backend/core/api/text_runtime.py 和 runtime_session.py，理解文本请求怎么进入会话。
  4. 接着看 core/soul_companion/runtime.py、agent_loop.py、prompt.md，理解 Agent 如何调用工具。
  5. 最后看 plugins/soul-companion/__init__.py，理解每个生活查询工具实际做什么。
  6. 如果研究语音，再看 useWebRTC.ts、webrtc.py、pipeline/factory.py。
  7. 如果研究记忆，再看 context_manager.py、sql_store.py、models.py。
