Soulbot Deployment Notes

Vercel + Railway 生产部署配置说明

本文记录 Soulbot 前端发布到 Vercel、后端发布到 Railway、以及 soulmeet.space / api.soulmeet.space 域名联通的实际配置。 重点是保留前端在 Vercel，后端 API 走 Railway。

前端 https://soulmeet.space 由 Vercel 托管。
后端 https://soulmeetweb-production.up.railway.app 已健康。
API 域名 api.soulmeet.space DNS 已指向 Railway。

当前结论

Railway 后端默认域名已通过健康检查；Vercel 主域 soulmeet.space 配置有效； Vercel DNS 中已经添加 Railway 要求的 CNAME 和 TXT 记录。

注意：刚改完 DNS 后，本机系统 DNS 可能仍缓存旧 Vercel 解析，普通 curl https://api.soulmeet.space/health 可能短时间仍返回 Vercel DEPLOYMENT_NOT_FOUND。使用 Cloudflare/Google DNS 查询时已经能看到新记录。

前端环境变量

Vercel 环境变量页存在 VITE_BACKEND_ORIGIN、 VITE_VOICE_TRANSPORT、Sentry 相关变量。当前线上构建产物中 VITE_BACKEND_ORIGIN 实际写入的是 Railway 默认域名： https://soulmeetweb-production.up.railway.app。

建议在 Vercel 环境变量页恢复加载后，将 VITE_BACKEND_ORIGIN 更新为 https://api.soulmeet.space，然后重新部署一次前端。Vite 会在构建时固化 VITE_* 变量，所以只改变量不重部署不会影响线上 JS。

DNS 记录

soulmeet.space 的 nameserver 是 Vercel： ns1.vercel-dns.com 和 ns2.vercel-dns.com。 因此 Railway 的自定义域名验证记录要在 Vercel 团队域名页里添加。

Type Name Value TTL
CNAME api pewzv134.up.railway.app 60
TXT _railway-verify.api railway-verify=2c7fc00df76f991a4f6e4915eb774a1b1ed721c6ca0c4aa0af73d27d990a9259 60

为什么不能只配 CNAME

Vercel DNS 原本有 * ALIAS cname.vercel-dns-017.com. wildcard， 所以 api.soulmeet.space 会被通配到 Vercel，导致访问 API 时返回 DEPLOYMENT_NOT_FOUND。Railway 的自定义域名还需要 TXT 验证，缺 TXT 时可能已经到 Railway Edge，但会返回 Application not found。

验证命令

dig @1.1.1.1 +short api.soulmeet.space CNAME
dig @1.1.1.1 +short _railway-verify.api.soulmeet.space TXT
curl -i https://soulmeetweb-production.up.railway.app/health
curl -i https://api.soulmeet.space/health

如果系统 DNS 仍缓存旧记录，可临时用最新 Railway IP 强制解析验证： 

dig @1.1.1.1 +short api.soulmeet.space A
curl --resolve api.soulmeet.space:443:69.46.46.91 -i https://api.soulmeet.space/health

成功结果应包含：

HTTP/2 200
server: railway-hikari

{"ok":true}

文本服务错误排查

聊天页提示「文本服务暂时没有连接上，请确认后端已启动后再试」时，不一定是 Railway 后端没有启动。前端代码会在两类情况显示同一句话：请求返回 500 以上状态码，或浏览器 fetch/EventSource 出现网络层 TypeError。

2026-06-15 复查时，后端健康检查、文字消息接口和 SSE 进度通道均已通过： 

curl -i https://api.soulmeet.space/health
curl -i https://api.soulmeet.space/api/text-runtime/message \
  -H 'Origin: https://soulmeet.space' \
  -H 'Content-Type: application/json' \
  --data '{"text":"你好","audience":"Aini","session_id":"diag-api-domain","timeout_seconds":5,"token":""}'
curl -i 'https://api.soulmeet.space/api/text-runtime/events/diag-sse?audience=Aini' \
  -H 'Origin: https://soulmeet.space'

如果这些命令返回 200，而浏览器仍显示该提示，优先处理浏览器缓存、DNS 缓存和 Vercel 构建变量：强制刷新页面，等待 DNS 缓存过期，确认 VITE_BACKEND_ORIGIN 已改为 https://api.soulmeet.space 后重新部署前端。

本次踩坑记录

  1. Vercel wildcard DNS 会让未显式配置的子域名全部落到 Vercel。
  2. Railway 自定义域名弹窗里同时给 CNAME 和 TXT，必须两条都加。
  3. CNAME 目标要用 Railway 弹窗给出的 pewzv134.up.railway.app，不能随手用默认服务域名。
  4. Vercel 添加 TXT 时会出现 Wildcard Domain Override 确认框，需要确认后才真正落表。
  5. DNS 权威记录生效后，本机系统 resolver 仍可能缓存旧 IP，排查时优先查 @1.1.1.1 或 @ns1.vercel-dns.com。
  6. Vite 前端环境变量在构建期写入，改 Vercel 环境变量后必须重新部署。
