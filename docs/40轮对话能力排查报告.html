40 轮对话能力排查报告

评估对象：Soulbot 当前架构（批次一、二已完成；批次三、四、五未实施） |  参照：docs/Harness稳定性改造计划与验收标准.md |  日期：2026-06-12
总体结论：不能直接说"可以"。 40 轮中约 18 轮现架构 + 已规划批次完成后可直接满足，约 17 轮存在依赖或风险， 5 轮（轮 23、32、33、34 及其衍生轮）涉及当前架构与改造计划都未覆盖的真实缺口。 另有 7 个轻量轮（≤3–5 秒）的性能要求与改造计划自身的延迟目标（boundary p95 < 5s）存在冲突。 这套 40 轮用例若现在实跑，预计在轮 5（延迟）、轮 14/17（跨任务槽位继承）、轮 23（目的地级联切换）、 轮 32–34（批量清理 / 跨任务聚合 / 机票误路由）处出现系统性失败。

一、五个结构性缺口（按严重程度排序）

缺口 1：跨任务级联失效 —— 改目的地后无法批量刷新关联证据

影响轮次：23、24、29（部分波及 25–30 的偏好迁移）

轮 23"改泉州，刷新厦门相关的天气、景点、美食、住宿证据，保留出发地/天数/偏好"， 需要的语义是：共享槽位（destination）变更 → 关联任务的旧 evidence 批量标记失效 → 偏好（known_inputs 中的非地点项）迁移到新上下文。 当前架构中 Frame / Trace 按任务隔离，任务之间没有"共享槽位"概念； 改造计划的 lifecycle 操作只有 pause / resume / cancel / pause_all / list_tasks， 五个批次中没有任何一项覆盖"槽位变更引发的证据级联失效"。 这是 40 轮中唯一需要新增架构设计的需求，不是实现工作量问题，而是语义模型缺位。

缺口 2：按主题批量取消 —— "把厦门相关的都不要了"无执行语义

影响轮次：32

当前 lifecycle_op=cancel 仅支持单个 lifecycle_target，pause_all 是全量操作。 "取消满足某条件的任务子集"（所有厦门相关任务，但保留泉州任务）介于两者之间： boundary 决策契约里 selected_task_ids 字段理论上能表达多目标， 但下游没有"对 selected_task_ids 逐个 cancel"的执行路径。补齐成本低（扩展 lifecycle 执行器即可），但目前没有。

缺口 3：跨任务证据聚合 —— 最终攻略无法整合车票/住宿/美食任务的结果

影响轮次：33、37、39

轮 33"整理最终版，包含动车、住宿、美食、路线"要求从多个独立任务（车票任务、住宿任务、美食任务）各自的 Trace/artifact 中取证据，合成一份方案。itinerary.compose 技能存在且定位正确（"把已有结果编排行程"）， 但它运行在单个任务的上下文里，没有跨任务 artifact 共享通道；FinalResponder 同样是单任务视角。 轮 37 的预算估算（汇总动车票价 + 住宿价格 + 餐饮）依赖同一机制。 需要设计"会话级证据视图"或"任务间 artifact 引用"，计划中未覆盖。

缺口 4：轻量轮延迟预算冲突 —— ≤3–5 秒的轮次结构性超时

影响轮次：5（≤3s）、18、22、26、36、40（≤5s）、31（≤8s 边缘）

这些轮的链路是：boundary LLM 判断 → 路由到闲聊 → 闲聊 LLM 生成回复，至少两次串行 LLM 调用。 改造计划给 boundary 的目标本身就是 p95 < 5 秒——意味着光路由这一步就可能吃掉全部预算。 要满足 ≤3 秒，必须做计划之外的事：boundary 换小模型/快模型、 或 boundary 与闲聊回复并行投机执行（闲聊回复先生成，boundary 判定为任务时丢弃）。 这不是缺陷，是计划目标与本测试要求之间的硬冲突，需要明确取舍。

缺口 5：机票拒绝路径缺失 + 关键词规则会主动误路由

影响轮次：34

要求：明确回答"暂不支持机票"，不能编造航班、不能误转动车。 EffectiveSkillCatalog 中确实没有机票技能，但当前没有显式的"识别为任务 → 无匹配能力 → 明确告知不支持"的回复路径， 最可能的实际行为是落入闲聊兜底由 LLM 自由发挥（编造风险）。 更直接的问题：goal_boundary.py 的 _task_kind_from_text 将含"票"的输入判为 train 类——"机票"包含"票"，确定性规则会把这一轮误绑到已有动车任务， 恰好命中本轮明令禁止的失败模式。这是此前评审指出的"关键词规则过拟合"的具体落点。

四、要让这 40 轮全部通过，需要补什么

  • 新增设计（计划外）：
    • 共享槽位与证据级联失效（缺口 1）：为任务引入"共享上下文槽位"（destination 等），变更时标记关联任务 evidence 为 stale 并保留偏好。这是最大的一块设计工作，建议作为"批次六"立项。
    • 会话级证据视图（缺口 3）：给 itinerary.compose / FinalResponder 提供跨任务 artifact 只读通道，支撑最终整合与预算估算。
    • 轻量轮快速通道（缺口 4）：boundary 用快模型，或与闲聊回复并行投机执行，否则 ≤3–5s 的 7 个轮次必然超时。
  • 小改动（一周内量级）：
    • lifecycle 执行器支持对 selected_task_ids 子集批量 cancel（缺口 2）。
    • 新增"任务意图但无匹配能力 → 显式不支持回复"路径，并修复 _task_kind_from_text 的"票"误匹配（缺口 5）——这也再次印证此前评审的结论：goal_boundary 的关键词规则应当删除而非扩充。
  • 先决条件：按计划完成批次三（状态机）和批次四（分支契约 + 规划收敛），它们是 40 轮中约 13 轮的前置依赖；批次五接线（生产用 SQLiteKVStore）虽不阻塞本测试，但为生产必需。

本报告为静态架构排查结论，未实跑 40 轮用例；逐轮判定基于当前代码 （goal_boundary.py、action_loop.py、runtime_session.py、技能包目录） 与改造计划文档的交叉核对。建议在批次三、四完成后将本表纳入 tests/harness_boundary_policy_stability_runner.py 的 case 机制实跑验证， 并注意：本表用例措辞不得进入生产代码的关键词规则，否则评测失去意义。
