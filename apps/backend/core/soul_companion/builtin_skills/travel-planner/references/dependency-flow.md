# Dependency Flow

本文件定义旅游规划的工具依赖关系。旅行规划开始后先读取本文件，再调用业务工具。

## 总体依赖图

第一批可并行：
- `weather` 查询目的地天气。
- 如果 origin 存在，或用户明确问车票/动车/高铁/跨城交通，调用 `train_tickets` 查询去程。
- 如果是一日游且 origin 存在，同一轮再调用一次 `train_tickets` 查询返程。
- 如果 days 大于 1，或用户提到住宿/酒店/住一晚，可同一轮调用 `local_search` 查询住宿。

第二批必须依赖天气：
- `local_search` 查询景点。天气返回前不要先查景点。
- 天气不适合户外时，先按室内、避雨、少走路、打车方便方向查；不要先查户外景点再补查室内。
- 高德 POI 更适合具体 POI 或短关键词，天气和老人条件由你拿到结果后筛选，不要全部塞进 `keyword`。
- 厦门雨天老人一日游快路径：优先 `place=厦门`、`category=scenic`、`keyword=厦门市博物馆 厦门科技馆 文化馆`、`max_results=5`。
- 如果首查结果明显不可用，再换一个具体 POI 或短关键词，例如 `中山路`；不要连续用多个复合关键词试探。
- 如果首轮景点结果里已有 1-2 个可用于行程的候选，优先筛选并进入美食查询；不要为了更有名、更完美继续补查。

第三批必须依赖景点或活动区域：
- `local_search` 查询美食。不要在第一批泛查城市级美食。
- `place` 优先用已选主景点、商圈、车站或午休区域；没有可用区域时才用 destination。
- 美食 `keyword` 也用短关键词，例如 `小吃`、`闽菜`、`沙茶面`、`清淡`，不要把“午餐、老字号、有座位、适合老人”等全部塞进一个长关键词。
- 如果餐厅结果里已有可用于午餐/晚餐的候选，优先筛选并收尾；不要为了更地道、更热门继续补查。

票价：
- 只有用户明确问多少钱、票价、价格、二等座/一等座价格时，才调用 `train_ticket_price`。
- 如果用户没有指定车次，只查推荐首选车次价格，不要查多趟车价格。

## 车次参数

去程 `train_tickets`：
- `departure` 用 origin。
- `destination` 用 destination。
- `date` 用用户日期或“明天”。
- `preference` 默认 `earliest`。
- `max_results` 用 15。

返程 `train_tickets`：
- `departure` 用 destination。
- `destination` 用 origin。
- `date` 用用户日期或“明天”。
- `preference` 用 `latest`。
- `max_results` 用 15。

同一路线同一天车次返回候选后，后续优先从已有结果筛选。只有工具失败、结果为空或用户追加新条件时，才再次查询同一路线同一天。
一日游返程查过 `latest` 后，不要再改成 `earliest` 重查；如果候选时间不满意，直接在已有返程候选中选较合适的，或说明可在 12306 再筛选更早班次。

## 补查原则

补查必须有明确理由：
- 工具失败。
- 结果为空。
- 结果全部与天气、老人少走路、饮食偏好冲突。
- 区域明显错误。
- 用户追加新条件。

“不够完美”“想找更有名”不是补查理由。已有结果足够支撑答复时，直接生成最终答复。
